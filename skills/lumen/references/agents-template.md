# Rule File Templates

Templates for the files generated by `/lumen rules`.

---

## AGENTS.md — Lumen section

This section is appended to `AGENTS.md`. If the file already contains other content,
add a separator and append at the end.

```markdown
## Lumen — Project Documentation

This project's documentation was generated and is maintained by **Lumen**, a skill
that acts as the persistent knowledge keeper for this repository.

**Skill location:** `skills/lumen/` (or `.claude/skills/lumen/` depending on setup)
**Skill trigger:** any `/lumen` command, or mentions of "lumen", "knowledge base",
"document the project", or broad architectural questions.

### Before starting any task

1. Read `AGENTS.md` for the project overview, tech stack, and documentation index.
2. Read `docs/high-level-design.md` for architecture and component map.
3. For component-specific work, read `docs/<component-name>/README.md`.
4. Check `docs/codestyle.md` for project patterns and code style.
5. Check `docs/rationale.md` for non-obvious decisions and their reasoning.
6. If the project has external integrations, check `docs/integrations.md`.

### Keeping documentation current

This documentation is not static — use Lumen to keep it in sync with the code.

| Command | Purpose |
|---------|---------|
| `/lumen scan` | Analyze code and update documentation |
| `/lumen update` | Quick sync from recent commits |
| `/lumen ingest` | Process raw files in docs/raw_data/ |
| `/lumen status` | Check coverage and freshness |
| `/lumen rules` | Regenerate these rule files |
| `/lumen <question>` | Query the documentation |

If you notice documentation is stale or incomplete, run `/lumen update` or
`/lumen scan` rather than editing docs manually — Lumen preserves manual edits
while keeping everything consistent.
```

## Cursor rules — .cursor/rules/agents/lumen.md

Create at `.cursor/rules/agents/lumen.md`:

```markdown
---
description: Project documentation — read before any task
globs:
alwaysApply: true
---

# Lumen — Project Documentation
```

## Symlink setup

After creating or updating `AGENTS.md`, ensure the symlink:

```bash
# If CLAUDE.md doesn't exist, or is a regular file that was already migrated:
ln -sf AGENTS.md CLAUDE.md
```

If `CLAUDE.md` was a regular file with existing content, its content has already
been migrated into `AGENTS.md` before creating the symlink.

---

## Handling pre-existing files

### Scenario: CLAUDE.md exists with content

1. Read current `CLAUDE.md` content
2. Create `AGENTS.md` with that content
3. Append Lumen section to `AGENTS.md`
4. Replace `CLAUDE.md` with symlink → `AGENTS.md`
5. Report: *"Migrated existing CLAUDE.md content to AGENTS.md and added Lumen section."*

### Scenario: CLAUDE.md is already a symlink

1. Check where it points
2. If it points to `AGENTS.md` → proceed with updating `AGENTS.md`
3. If it points elsewhere → ask the user how to handle it
4. Report what was done

### Scenario: AGENTS.md exists with content

1. Check if Lumen section already exists (search for "## Lumen")
2. If not present: append Lumen section
3. If present: update it
4. Ensure `CLAUDE.md` symlink exists
5. Report: *"Updated Lumen section in existing AGENTS.md."*

### Scenario: Both exist as regular files

1. Merge content: AGENTS.md as base, append any unique content from CLAUDE.md
2. Append Lumen section
3. Replace CLAUDE.md with symlink
4. Report: *"Merged AGENTS.md and CLAUDE.md, added Lumen section."*

### Scenario: .cursor/rules/ already exists

1. Check if `agents/lumen.md` exists
2. If yes: update content
3. If no: create `.cursor/rules/agents/` if needed, then create `lumen.md`
4. Do not modify other files in `.cursor/rules/`
